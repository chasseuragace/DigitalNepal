## What this actually means in practice

On the NID API: the entire system is a carefully scoped read-only wrapper. The government never needs to rebuild its existing NID database — it just needs to build a consent-controlled API layer on top of it. The architectural principle is that a bank asking "is this person who they say they are?" should get only a yes/no, while a loan origination system asking "give me this person's demographics for a loan application" needs the citizen to explicitly approve that specific request with a time-limited token. This consent architecture is what separates a useful identity system from a surveillance system, and it's exactly what Aadhaar got badly wrong in its early years (and had to retrofit). Nepal has the advantage of building it right from the start.

The `data_freshness` field in the `kyc_basic` response is more important than it looks. A lot of NID records in Nepal were captured years ago and addresses are outdated. Surfacing when the data was last updated lets the consuming application decide whether to ask the customer to verify their address separately — rather than blindly trusting stale data for a loan decision.

On the NRB sandbox: the three-tier structure is the key design decision. Tier 1 requiring zero approval means a developer can start testing international payment flows the same day they sign up — no waiting for NRB. This is what Pakistan's SBP sandbox got right. Tier 2's NPR 5 million monthly cap is high enough to be a real business but low enough that NRB can monitor for AML risk without needing to scale their team immediately. The connection between the two systems — using NID API for KYC verification at sandbox onboarding — means both initiatives reinforce each other from day one.

The risks section deserves emphasis on one point above all others: procurement capture. Both systems combined are roughly a NPR 200 million contract opportunity. Nepal's IT procurement history has significant problems with inflated tenders, politically connected firms without technical capability, and no accountability for delivery failure. The mitigation — mandate open-source stack with public GitHub repository, public code reviews, and evaluation committees with private sector technical members — needs to be written into the tender specifications before any contractor is engaged. If the code isn't public, the system won't be trusted and won't be audited, which defeats the purpose entirely.
